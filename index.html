迷宮生成與尋路系統

以 Python + Pygame 實作完整互動式迷宮遊戲，整合隨機迷宮生成動畫、 三種尋路演算法視覺化對決、道具怪物等遊戲機制，以及動態地圖縮放。

Python 3.12 Pygame DFS Maze Generation A* Pathfinding BFS · DFS Search OOP Design Real-time Animation

專案動機

為了深入理解圖論演算法的實際差異，我選擇以「迷宮」作為載體——將 DFS、BFS、A* 三種演算法放在同一場景中讓玩家即時切換觀察， 讓抽象的時間複雜度變成可視化的格子數與毫秒數。

演算法對決 MODULE 01

按 1 / 2 / 3 即時切換 DFS、BFS、A* 三種尋路策略。 HUD 即時顯示「探索格數」與「耗時 ms」，讓玩家親眼見證 A* 以曼哈頓距離啟發函數大幅減少冗餘搜索的威力。 再按相同鍵可關閉，支援任意起點重算（每次移動後同步更新）。

heapq priority queue came_from backtrack F = G + H real-time HUD
🎬
生成動畫 MODULE 02

將 DFS 迷宮生成的 while 迴圈拆解為單步 step()， 配合 Pygame 主迴圈以每秒 20 步的節奏渲染。 Stack 中的格子顯示黃色，Stack 頂端最亮； 遇到死路 pop 時黃色格子逐格退回，直觀呈現 Backtracking。 按 Enter 可跳過動畫。

step() generator pattern dt accumulator stack_set O(1) lookup
💣
道具 & 怪物 MODULE 03

迷宮中隨機散佈 3 顆炸彈，玩家踏上自動撿起，按 B 爆破四鄰牆壁（不可破壞邊界）。巡邏怪物生成於距出發點 曼哈頓距離 > max(row,col)/3 的通道， 每 0.25 秒隨機移動，碰觸玩家觸發 CAUGHT 結局。 炸牆後演算法路徑自動重算。

dt timer pattern SRCALPHA overlay Manhattan distance spawn
📐
動態尺寸縮放 MODULE 04

按 + / − 每次調整 10 格（11×11 ↔ 101×101）。 Cell 大小由 WIN_W ÷ max(rows, cols) 動態計算， 101×101 時格子縮至 5 px 仍可玩。視窗呼叫 set_mode() 重建，不需重啟程式。 奇數邊長由程式自動修正以確保 DFS 生成正確。

auto cell-scaling set_mode() hot-reload odd-dimension guard
演算法比較分析
演算法 搜索策略 保證最短路？ 時間複雜度 核心資料結構 啟發函數 實測（21×21）
▶ DFS 深度優先，沿路走到底 ✗ 否 O(V + E) list (stack) — 探索格多、耗時低
▶ BFS 廣度優先，層層擴散 ✓ 是 O(V + E) deque (queue) — 探索格最多、路徑最短
▶ A* 最佳優先，F = G + H 導引 ✓ 是 O(V log V) heapq (min-heap) 曼哈頓距離 探索格最少、效率最高

學習反思

這個專案讓我體會到「同樣是圖搜索，資料結構的選擇決定了一切」。 DFS 用 list.pop()，BFS 改用 deque.popleft()， A* 引入 heapq——三種演算法的差異在視覺化後變得一目了然。 動畫生成部分將迴圈拆解為 step() 的設計，讓我理解了 「遊戲迴圈友善的生成器模式」，這個思維可遷移至任何需要逐幀 處理的任務。
